## 5. Data Model

### `Order` Entity (`com.example.orderservice.model.Order`)
The `Order` entity represents a customer's order. It is mapped to the `orders` table in the PostgreSQL database.

| Field | Type | Description | Constraints |
|---|---|---|---|
| `id` | `Long` | Unique identifier for the order. | Primary Key, Auto-generated |
| `customerId` | `Long` | Identifier of the customer who placed the order. | Not Null |
| `itemIds` | `List<String>` | List of product SKUs or identifiers included in the order. | ElementCollection |
| `totalAmount` | `BigDecimal` | The total monetary value of the order. | Not Null, Precision |
| `status` | `OrderStatus` | Current status of the order. | Not Null, Enum (String) |
| `createdAt` | `LocalDateTime` | Timestamp when the order was created. | Not Null |
| `updatedAt` | `LocalDateTime` | Last timestamp when the order was updated. | Not Null |

### `OrderStatus` Enum (`com.example.orderservice.model.OrderStatus`)
Defines the possible states an `Order` can be in.

- `PENDING`: Order has been placed but not yet processed.
- `PROCESSING`: Order is being prepared for shipment.
- `SHIPPED`: Order has been dispatched.
- `DELIVERED`: Order has been successfully delivered to the customer.
- `CANCELLED`: Order has been cancelled.

### Spring Profiles and Their Effects
Spring profiles allow for environment-specific configuration.
- **`dev`**: Default profile for local development. Typically uses an embedded database (H2) or local Dockerized services (PostgreSQL, Kafka). Logging is usually set to DEBUG.
- **`test`**: Used during integration and end-to-end testing. Often uses Testcontainers for isolated and ephemeral dependencies (PostgreSQL, Kafka). May activate specific test-related configurations.
- **`prod`**: Production environment profile. Configured with external, highly available services (managed databases, Kafka clusters). Optimized for performance and security. Logging is typically INFO/ERROR.
- **`docker`**: Specific profile for running inside Docker containers. Might use hostnames (e.g., `host.docker.internal` or service names `kafka:9092`, `postgres:5432`) instead of `localhost` for service discovery.

To activate a profile, use `spring.profiles.active` property or the `--spring.profiles.active` CLI argument.

### Environment Variables
Environment variables can override `application.properties`. Common ones include:
- `SPRING_DATASOURCE_URL`
- `SPRING_DATASOURCE_USERNAME`
- `SPRING_DATASOURCE_PASSWORD`
- `SPRING_KAFKA_BOOTSTRAP_SERVERS`
- `SERVER_PORT`
- `SPRING_PROFILES_ACTIVE`
